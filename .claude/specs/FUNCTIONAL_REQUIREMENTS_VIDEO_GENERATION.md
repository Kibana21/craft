## 1. Flow Overview

The video generation flow is a four-stage pipeline. Each stage produces an output that feeds the next. 
This workflow is to be trigerred once Create reel / video artifact is triggered from inside a project.

```
Presenter Definition
        │
        ▼
  Script Creation
        │
        ▼
 Scene Storyboarding  ←──── (uses Presenter + Script)
        │
        ▼
  Video Generation   ←──── (uses all Scenes, each with embedded Presenter)
        │
        ▼
   Playback / Download
```

**The critical design principle throughout is presenter consistency:** the presenter's appearance is defined once, embedded into every scene at storyboard generation time, and cannot be overridden per scene. This guarantees the same person appears in every scene of the generated video.

## 2. Presenter Definition

### 2.1 Purpose

A presenter is the person who will appear on screen throughout the video. Defining a presenter accurately is the most important input for visual quality and consistency — the AI video model uses the presenter description as a prompt for how to render the human subject in every scene.

### 2.2 Presenter Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| Name | Text | Yes | The presenter's name or role label (e.g. "Sarah", "Insurance Advisor") |
| Age Range | Dropdown | Yes | Approximate age range of the presenter |
| Appearance Keywords | Text | Yes | Short descriptive keywords (e.g. "South Asian, professional, dark hair, business suit") |
| Full Appearance Description | Long text | Yes | A detailed prose paragraph describing the presenter in full — this is what gets injected into video prompts |
| Speaking Style | Single-select | Yes | The presenter's communication style (see §2.3) |

### 2.3 Speaking Style Options

| Value | Description |
|-------|-------------|
| Authoritative | Confident, direct, credible — suits compliance, leadership, or expert-led content |
| Conversational | Natural, relaxed, approachable — suits training, onboarding, or customer-facing content |
| Enthusiastic | High energy, motivating, upbeat — suits campaign, product launch, or promotional content |
| Empathetic | Warm, caring, supportive — suits healthcare, financial advisory, or sensitive topics |

### 2.4 Full Appearance Description

This is the most critical presenter field. It is a detailed, natural-language paragraph that describes:
- Physical appearance (skin tone, hair colour and style, eye colour, build)
- Clothing and professional attire (colour, style, formality level)
- Setting or background context (if relevant)
- Demeanour and manner (posture, expression, energy)

**Example:**
> "A South Asian woman in her early 30s with long dark hair tied neatly back. She wears a navy blue tailored blazer over a light grey blouse, projecting a professional and approachable manner. She speaks directly to camera with a composed, confident expression. The background is a soft-focus modern office with warm ambient lighting."

This paragraph is inserted verbatim into the video generation prompt for every scene, ensuring visual consistency of the presenter throughout the entire video.

### 2.5 AI-Assisted Appearance Generation

The appearance description can be written manually or generated by AI.

**How it works:**
1. The user fills in the Appearance Keywords and Speaking Style fields
2. The user clicks "Generate from keywords"
3. The AI reads the keywords and speaking style and generates a detailed, well-structured appearance paragraph
4. The paragraph is placed into the Full Appearance Description field
5. The user can accept it, edit it, or regenerate

The AI produces a more complete and prompt-optimised description than most users would write manually, which typically results in better video output quality.

### 2.6 Presenter Library (Reusability)

Presenters are saved to a shared library and can be reused across multiple projects or video sessions.

- When creating a new video session, the user can select a previously saved presenter
- Selecting a saved presenter pre-fills all presenter fields
- The user can modify the fields for a specific session without permanently changing the saved presenter (unless they explicitly save changes back)
- A presenter can be linked to any number of projects or video sessions

### 2.7 Presenter Lock After Scene Generation

Once scenes are generated using a presenter, the presenter's Full Appearance Description is embedded into each scene's generation prompt. At this point:
- The presenter cannot be swapped out without regenerating all scenes
- The appearance description on individual scene prompts cannot be overridden
- All scenes are tagged "Presenter Locked" to reflect this

This lock is intentional — it guarantees visual consistency across the entire video.

## 3. Script Creation

### 3.1 Purpose

The script is the full narration or dialogue that the presenter will deliver. It drives the content of every scene — the AI uses the script to determine how many scenes to create, what is said in each, and how long each scene should be.

### 3.2 Script Field

A single large free-text field for the full video script. There is no per-scene script input at this stage — the AI splits the unified script into scenes in the next step.

### 3.3 Target Duration

A duration setting (associated with the project or video session) informs how long the video should be. Available options: 30 seconds, 60 seconds, 90 seconds, 2 minutes, 3 minutes, 5 minutes.

The target duration is used by the AI both when drafting a script (to produce appropriately sized content) and when splitting the script into scenes (to determine how many scenes are needed and how much content each scene should carry).

### 3.4 Word Count & Duration Estimate

The system provides live feedback while the user writes:
- Current word count
- Estimated spoken duration, calculated at a standard speaking pace

This helps the user align the script length to the target duration before proceeding to scene generation.

### 3.5 AI Script Drafting

The user can ask the AI to write a complete script from scratch based on the project brief.

**Inputs used by the AI:**
- Video type (product explainer, training, testimonial, etc.)
- Target audience
- Target duration
- Key message
- Brand kit / tone
- Call-to-action text

**Output:**
- A complete, structured script covering introduction, key content, and a closing CTA
- Written to fit approximately within the target duration
- Appropriate in tone for the video type and audience

The generated script fills the script field and can be freely edited.

### 3.6 Tone Rewrite

Four tone adjustments can be applied to the current script at any time:

| Tone Option | What it does |
|-------------|-------------|
| Warm & Personal | Rewrites in a friendly, human, empathetic register |
| More Professional | Adjusts to a formal, polished, corporate register |
| Shorter | Condenses to the essential points while retaining structure |
| Stronger CTA | Strengthens the closing section; makes the call-to-action more compelling and urgent |

Each tone rewrite:
1. Sends the full current script and the selected tone to the AI
2. Replaces the script field with the rewritten version
3. Saves the previous script as a version (see §3.7)

The rewritten script can be further edited or re-rewritten with a different tone.

### 3.7 Script Version History

Every AI action (draft or tone rewrite) automatically saves the previous script as a version. Versions record:
- The full script text at that point in time
- Which action created the version (draft, warm rewrite, professional rewrite, shorter, stronger CTA)
- The date and time

This allows users to recover a previous version of the script if a rewrite moves in the wrong direction.

### 3.8 Storyboard Staleness

The system tracks which script version was used when scenes were last generated. If the script is updated after scenes have been generated, the storyboard step displays a warning: scenes were generated from an older version of the script and may not match the current script. The user is prompted to regenerate.

## 4. Scene Storyboarding

### 4.1 Purpose

Scene storyboarding converts the linear script into a structured set of production-ready scenes. Each scene becomes one video clip in the final output.

### 4.2 Automatic Scene Generation

When the user arrives at the storyboard step, the system checks whether valid scenes exist:

- **No scenes exist** → AI scene generation runs automatically
- **Scenes exist but script has been updated** → A staleness warning is shown with a prompt to regenerate
- **Scenes exist and script is unchanged** → Existing scene cards are shown as-is

### 4.3 What Scene Generation Produces

The AI analyses the script and target duration to split the content into logical scenes. For each scene, the AI produces:

| Output | Description |
|--------|-------------|
| Scene name | A short descriptive title (e.g. "Introduction", "Product Benefits", "Closing CTA") |
| Dialogue | The specific lines of the script assigned to this scene |
| Setting | A description of the visual environment (e.g. "Modern office with large windows", "Outdoor park, daytime") |
| Camera framing | Recommended shot composition (see §4.5) |

The number of scenes generated is proportional to the target duration and the natural break points in the script.

### 4.4 Scene Cards

Each scene is displayed as a card in the sequence order it will play in the video. Cards are numbered starting from 1.

Every scene card shows and allows editing of:
- **Scene number** — fixed; reflects sequence position
- **Scene name** — editable text field
- **Dialogue** — editable; shown in italic style with a left accent border to distinguish it from metadata
- **Setting** — editable text field describing the visual environment
- **Camera framing** — editable dropdown
- **"Presenter Locked" tag** — non-editable; confirms the presenter appearance is embedded
- **"Brand Locked" tag** — non-editable; confirms the brand kit is embedded
- **Save button** — saves edits to this specific card

### 4.5 Camera Framing Options

| Option | Description |
|--------|-------------|
| Wide Shot | Full environment visible; presenter small in frame — establishes context |
| Medium Shot | Presenter from waist up — standard interview or presentation framing |
| Close-Up | Face and shoulders — emphasises emotion, direct address |
| Over the Shoulder | Camera behind one person looking at another — dialogue or interaction scenes |
| Two Shot | Two people in frame — conversation or paired presenter format |
| Aerial / Drone | Top-down or elevated view — location establishing shots, scale |
| POV | First-person perspective — immersive, from the viewer's point of view |

### 4.6 Editing Scenes

All fields on a scene card are editable inline. The user clicks into any field, makes changes, and clicks the card's **Save** button. The button shows "Saved ✓" briefly to confirm success. Scene edits do not automatically save — the Save button must be clicked.

Editing a scene's dialogue or setting does not trigger AI regeneration of that scene. The merged prompt used for video generation is rebuilt from the saved field values when video generation is triggered.

### 4.7 Inserting a Scene

Users can add a new scene at any position in the sequence:

- **At the start** — inserts a blank scene before Scene 1
- **Between two existing scenes** — a small insert button between adjacent cards opens an insert modal

**Insert scene modal fields:**
- Scene name
- Dialogue
- Setting
- Camera framing

On confirmation, the new scene is inserted at the chosen position. All scenes after the insertion point are automatically renumbered to maintain a contiguous sequence.

### 4.8 Deleting a Scene

A delete button on each scene card removes that scene permanently. The remaining scenes are automatically renumbered. Deletion does not require a separate confirmation step on the scene card.

### 4.9 Regenerating All Scenes

The "Regenerate all scenes" button discards all current scenes and re-runs AI scene generation from the current version of the script. This is appropriate when:
- The script has changed significantly
- The existing scenes do not reflect the intended structure
- The user wants a fresh AI split with a different approach

Regeneration replaces all scenes — manually edited scenes are lost if not saved elsewhere.

### 4.10 Scene Sequence

Scenes are always maintained in a contiguous integer sequence starting at 1. The sequence drives:
- The order scenes appear in the storyboard view
- The order scenes are generated by the video AI
- The visual continuity chain (each scene extends from the previous)

## 5. The Merged Scene Prompt

### 5.1 What It Is

The merged scene prompt is the complete instruction sent to the AI video model for each scene. It is built automatically at scene generation time and stored with each scene. It is not directly visible to the user but drives all video output.

### 5.2 How It Is Built

Each scene's merged prompt concatenates the following components:

```
{scene dialogue}

Setting: {scene setting}
Camera: {scene camera framing}

Presenter: {presenter.full_appearance}

Speaking style: {presenter.speaking_style}

Brand kit: {project.brand_kit}
```

### 5.3 Why It Matters

- **Presenter consistency**: The same full_appearance text appears in every scene's prompt. This is what causes the AI to render the same-looking person in every clip.
- **Brand consistency**: The brand kit style instruction is repeated in every scene's prompt, ensuring visual tone stays consistent.
- **Self-contained**: Each scene prompt contains all the information the video AI needs — there is no cross-referencing between scenes at generation time. Each scene prompt is a standalone production instruction.

### 5.4 Regeneration

The merged prompt is rebuilt whenever:
- Scenes are regenerated via "Regenerate all scenes"
- A new scene is inserted (its prompt is built at creation time)

If the presenter is changed after scenes are generated, the existing merged prompts are **not** automatically updated. Scenes must be regenerated to pick up the new presenter description.

## 6. Video Generation

### 6.1 Triggering Generation

Video generation is triggered by the user from the project interface (e.g. a "Generate Video" button). At the point of triggering:
- At least one scene must exist
- Scenes must have dialogue and a merged prompt built
- No other video generation job for this project can be active (see §6.3)

### 6.2 What Happens on Trigger

1. A new video record is created with status **Queued** and progress at 0%
2. A background job is dispatched to the video generation worker
3. The video card appears in the UI immediately showing the Queued state
4. The system begins polling the video status every 5 seconds

### 6.3 One Active Job Per Project

Only one video generation job can be active (Queued or Rendering) per project at any time. If a job is already active when the user attempts to generate again, the system rejects the request with an error: "A video is already being generated for this project." The user must wait for the current job to complete or fail before starting a new one.

### 6.4 Version Numbering

Each generated video is assigned an incremental version number within the project: Version 1, Version 2, Version 3, and so on. Previous versions are not deleted when a new one is generated.

### 6.5 Scene-by-Scene Generation Process

The video is generated one scene at a time, in sequence order:

**Scene 1 — Independent generation:**
- The AI video model generates Scene 1 using only Scene 1's merged prompt
- No prior video context is provided

**Scenes 2 through N — Extension chaining:**
- Each subsequent scene is generated by extending the previous scene's video output
- The AI receives: the current scene's merged prompt + the video clip from the previous scene
- This extension mechanism preserves visual continuity — the presenter's appearance, environment lighting, and style carry forward naturally from scene to scene

This chaining approach is what ensures the final video looks like one cohesive piece rather than a collection of disconnected clips.

### 6.6 Progress Tracking

After each scene is generated, the system updates:
- `progress_percent` — a percentage from 0 to 100 based on scenes completed vs total scenes
- `current_scene` — the scene number that was just completed

Example for a 4-scene video:
- After Scene 1: 25% complete, "Scene 1 rendering"
- After Scene 2: 50% complete, "Scene 2 rendering"
- After Scene 3: 75% complete, "Scene 3 rendering"
- After Scene 4: 100% complete → status transitions to Ready

### 6.7 Completion

When all scenes have been generated:
1. The final video clip (the fully extended output from the last scene) is saved as an `.mp4` file
2. The video record is updated: status → **Ready**, file URL set, progress → 100%
3. The video card updates to show a play button and download option

### 6.8 Failure Handling

If an error occurs during generation (at any scene):
1. The video status is set to **Failed**
2. The error message from the AI video service is stored on the video record
3. The video card displays the failure status and shows the error message text
4. The user can attempt to generate a new video (a new Version N is created — failed videos are not retried in place)

**Common failure causes:**
- Scene prompt too long or contains policy-violating content
- AI video model quota exceeded
- Operation timeout (generation took longer than the maximum wait time)
- Empty or malformed model response

### 6.9 Polling Behaviour

The frontend polls the video status endpoint every 5 seconds while any video for the project is in a Queued or Rendering state. Polling stops automatically when all videos reach a terminal state (Ready or Failed). No user action is needed to see progress updates.

## 7. Video Playback & Download

### 7.1 Playing a Video

Once a video has status **Ready**, clicking the video thumbnail opens a full-screen video player overlay.

**Player features:**
- Auto-plays on open
- Standard playback controls: play/pause, seek bar, volume, fullscreen toggle
- Video title shown in the header
- Close button (× in the corner)
- Clicking outside the player area also closes it
- Download button in the footer

### 7.2 Downloading a Video

Ready videos can be downloaded as `.mp4` files from two places:
1. The download button on the video card in the project view
2. The download button inside the full-screen player overlay

The downloaded file is named using the video's version title (e.g. `Version 1.mp4`).

### 7.3 Streaming

Videos are served via a streaming endpoint rather than a direct file download link. This means:
- Videos begin playing before the full file is loaded
- The player can seek to any position without waiting for the full file to buffer
- The same URL is used for both in-browser playback and file download

## 8. Video Management

### 8.1 Video List

All videos generated for a project are listed in the Videos section, ordered from newest to oldest. Each version is shown as a separate card.

### 8.2 Video Card — States

Each video card displays different content depending on the video's current status:

**Queued:**
- Hourglass icon
- "Queued…" label
- No progress bar yet

**Rendering:**
- Progress bar (fills left to right as scenes complete)
- Percentage complete (e.g. "50% complete")
- Current scene label (e.g. "Scene 2 rendering")

**Ready:**
- Play button thumbnail (clickable — opens player)
- Version title
- Ready status badge
- Download button
- Delete button

**Failed:**
- Red ✕ icon
- "Failed" label
- Error message text shown below the card title
- Delete button

### 8.3 Deleting a Video

Any video can be deleted regardless of status. The delete flow:
1. First click: the delete button turns red and shows "Confirm delete?"
2. A "Cancel" link appears alongside the confirm button
3. Second click on "Confirm delete?": the video is permanently deleted
4. Clicking "Cancel" resets the button to its initial state

Deletion removes the video record and the associated `.mp4` file (if the video was in Ready state). Deleting a Queued or Rendering video cancels the job. Deletion is irreversible.

### 8.4 Multiple Versions

Previous video versions are not replaced when a new generation is triggered. All versions remain accessible in the video list. The user may compare versions, download different versions, or delete old ones individually.

There is no version promotion or "mark as final" concept — all versions are equal from a system perspective.

## 9. AI Assist Features

### 9.1 Generate Presenter Appearance

| Attribute | Detail |
|-----------|--------|
| Where | Presenter step |
| Trigger | "Generate from keywords" button |
| Input | Appearance keywords + speaking style |
| Output | A detailed prose paragraph describing the presenter's appearance, clothing, and manner |
| Editable after | Yes — the result fills a text field that the user can modify |
| Saved as | The presenter's Full Appearance Description |
| Impact on video | This paragraph is injected into every scene's merged prompt |

**Guidance for good output:** The more specific the appearance keywords, the more consistent and accurate the AI-generated description. Vague keywords (e.g. "professional") produce generic descriptions. Specific keywords (e.g. "40s, East Asian, glasses, navy suit, white shirt, short hair, calm expression") produce precise, prompt-optimised descriptions that result in a more consistent rendered presenter.

---

### 9.2 Draft Script from Brief

| Attribute | Detail |
|-----------|--------|
| Where | Script step |
| Trigger | "Auto-draft from brief" button |
| Input | Video type, target audience, target duration, key message, brand kit, CTA text |
| Output | A complete structured script |
| Editable after | Yes |
| Version saved | Yes — previous script saved as a version automatically |

---

### 9.3 Tone Rewrite

| Attribute | Detail |
|-----------|--------|
| Where | Script step |
| Trigger | One of four tone chips |
| Input | Full current script + selected tone |
| Output | Full rewritten script in the selected tone |
| Editable after | Yes |
| Version saved | Yes — previous script saved as a version automatically |
| Tones available | Warm & Personal, More Professional, Shorter, Stronger CTA |

---

### 9.4 Split Script into Scenes

| Attribute | Detail |
|-----------|--------|
| Where | Storyboard step |
| Trigger | Automatic on arrival (if no scenes), or "Regenerate all scenes" button |
| Input | Full script + target duration |
| Output | Ordered set of scenes, each with name, dialogue, setting, and camera framing |
| Editable after | Yes — every scene field is editable |
| Overwrites | All existing scenes for the project |

---

### 9.5 AI Behaviour Rules (All Features)

- While an AI call is in progress, the triggering button is disabled to prevent duplicate submissions
- If an AI call fails, the existing content in the field is preserved unchanged; an error message is displayed near the button
- AI-generated content is not applied until the user accepts it (via continuing to the next step or clicking Save on a scene card)
- The AI does not have memory between calls — each call is independent

## 10. Business Rules & Constraints

### 10.1 Presenter Required for Scene Generation

Scenes can be generated without a presenter — however, the merged prompt will not include a presenter appearance description. This results in AI video output with an unspecified or inconsistent human subject. For best results, a presenter should be defined before scenes are generated.

### 10.2 Script Required for Scene Generation

Scene generation requires a non-empty script. The AI cannot split scenes from blank content.

### 10.3 Scenes Required for Video Generation

At least one scene must exist before video generation can be triggered. Each scene must have a dialogue value to produce a meaningful merged prompt.

### 10.4 One Active Video Job Per Project

Only one video generation job (Queued or Rendering) is allowed per project at any time. This prevents:
- Redundant API calls to the video model
- Conflicting progress updates
- Wasted compute resources

### 10.5 Scene Order is Immutable During Generation

The scene sequence is fixed at the moment video generation is triggered. Reordering, adding, or deleting scenes during an active generation job has no effect on the job in progress. If the user changes scenes after triggering generation, those changes will only be reflected in the next generation job.

### 10.6 Presenter Changes Do Not Retroactively Update Scenes

If the presenter's Full Appearance Description is updated after scenes have been generated, the existing scenes retain their original embedded description. To apply the updated presenter across all scenes, the user must regenerate scenes.

### 10.7 Maximum Poll Wait Time

The video generation system will wait a maximum of 15 minutes (900 seconds) per scene before timing out. If a scene does not complete within this time, the job is marked as Failed with a timeout error message.

### 10.8 Scene Count and Duration Alignment

The number of scenes generated by the AI is proportional to the target duration. Longer target durations produce more scenes. As a rough guide:
- 30-second video → 1–2 scenes
- 60-second video → 2–3 scenes
- 90-second video → 3–4 scenes
- 2-minute video → 4–5 scenes
- 3-minute video → 5–7 scenes
- 5-minute video → 8–12 scenes

## 11. States & Transitions

### 11.1 Video Status State Machine

```
                    ┌─────────────────────────┐
                    │  Generation Triggered   │
                    └────────────┬────────────┘
                                 │
                                 ▼
                          ┌──────────┐
                          │  QUEUED  │  ← Job dispatched; worker not yet started
                          └────┬─────┘
                               │  Worker picks up job
                               ▼
                        ┌────────────┐
                        │ RENDERING  │  ← Scenes being generated one by one
                        └──────┬─────┘
                    ┌──────────┴──────────┐
                    │                     │
                    ▼                     ▼
               ┌─────────┐          ┌────────┐
               │  READY  │          │ FAILED │
               └─────────┘          └────────┘
           (video file saved)    (error stored)
           (play + download)     (error shown)
                    │                     │
                    └──────────┬──────────┘
                               │
                               ▼
                          (deleteable)
```

### 11.2 Scene Lifecycle

```
Script exists
     │
     ▼
AI splits script → Scenes created (sequence 1, 2, 3...)
     │
     ├── User edits inline → Scene saved (unchanged merged_prompt until regeneration)
     ├── User inserts scene → New scene added at position N; subsequent scenes renumbered
     ├── User deletes scene → Scene removed; subsequent scenes renumbered
     └── User regenerates → All scenes deleted and replaced with new AI split
          │
          ▼
     Video generation uses scenes in sequence order
          │
          ├── Scene 1 → Independent Veo generation
          ├── Scene 2 → Extends Scene 1 output
          ├── Scene 3 → Extends Scene 2 output
          └── Scene N → Extends Scene N-1 output → Final MP4 saved
```

### 11.3 Presenter Consistency Lock

```
Presenter defined (appearance description written)
     │
     ▼
Scenes generated → Presenter appearance embedded in each scene's merged_prompt
     │
     ▼
"Presenter Locked" tag shown on all scene cards
     │
     ├── Presenter updated in library?
     │        └── Scenes NOT automatically updated — stale until regenerated
     └── Scenes regenerated → New appearance description embedded in new scenes
```

*This document describes the video generation pipeline as implemented in April 2026. Scene extension chaining is specific to the Google Veo 3.1 API (`veo-3.1-generate-001`). The polling interval, maximum wait time, and scene count heuristics are configurable system parameters.*
